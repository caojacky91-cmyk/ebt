About

ELIZA is a natural language conversation program described by Joseph Weizenbaum in January 1966 [1].
It features the dialog between a human user & a computer program representing a mock Rogerian psychotherapist.
The original program was implemented on the IBM 7094 of the Project MAC time-sharing system at MIT & was written in MAD-SLIP.

This is how Joseph Weizenbaum discussed his choice for a conversation model as it would be found in psychotherapist's session:
At this writing, the only serious ELIZA scripts which exist are some which cause ELIZA to respond roughly as would certain psychotherapists (Rogerians). ELIZA performs best when its human correspondent is initially instructed to "talk" to it, via the typewriter of course, just as one would to a psychiatrist. This mode of conversation was chosen because the psychiatric interview is one of the few examples of categorized dyadic natural language communication in which one of the participating pair is free to assume the pose of knowing almost nothing of the real world. If, for example, one were to tell a psychiatrist "I went for a long boat ride" & he responded "Tell me about boats", one would not assume that he knew nothing about boats, but that he had some purpose in so directing the subsequent conversation. It is important to note that this assumption is one made by the speaker. Whether it is realistic or not is an altogether separate question. In any case, it has a crucial psychological utility in that it serves the speaker to maintain his sense of being heard & understood. The speaker furher defends his impression (which even in real life may be illusory) by attributing to his conversational partner all sorts of background knowledge, insights & reasoning ability. But again, these are the speaker's contribution to the conversation.
For a conversation example given by Joseph Weizenbaum in his article see the Eliza Test page.
 

About e.js

"e.js" is an object oriented JavaScript library for [multiple] instances of the Eliza program.

Synopsis:

         new E( <random-choice-disable-flag> ); // default 'false'
         E.prototype.t( <inptstr> );
         E.prototype.I();
         E.prototype.F();
         E.prototype.x();

Usage:

         var eliza = new E();
         var initial = eliza.I();
         var reply = eliza.t(inptstr);
         if (eliza.q) {
             // last user input was a quit phrase
         }

         // method `t()' returns a final phrase in case of a quit phrase
         // but you can also get a final phrase with:
         var final = eliza.F();

         // other methods: reset memory & internal state
         eliza.x();
         // to set the internal memory size override property `W':
         eliza.W = 100; // (default: 20)

         // to reproduce the example conversation given by J. Weizenbaum
         // initialize with the optional random-choice-disable flag
         var originalEliza = new E(true);

As E is a totally self-contained object & instances use their own internal memory it's possible to have multiple instances of the E object talking to each other.

E is also a general chatbot engine that can be supplied with any rule set.

Data Structures (cf: "d.js"):
         
         eI ......... Array of init phrases
         eF  .......... Array of final phrases
         eQ ............ Array of quit phrases
         eX ............. Array of alternating name value pairs for preprocsing
         eP ............ Array of alternating name value pairs for pstprocsing
         eS ........... Object of wrds & their synonyms (as array)
         eT ... rgxp/replacement pairs to be performed as final cleanings
         eK ......... Array of kws with decomps & reasmbs

         eK elements are of:
       
         ["<key>", <rank>,
           [
             [ // first decomp pattern
               "<decomp>",
               ["<reasmb>", "<reasmb>", "<reasmb>"]
             ],
             [ // second decomp pattern
               "<decomp>",
               ["<reasmb>", "<reasmb>", "<reasmb>"]
             ]
           ]
         ]
       
         use "<key|key...>" to add OR statements.

ks with higher rank take precedence. decomps are matched in definition order.
         reasemblies are chosen by random or cycled through if the no-random flag ist set.
 
         the special kw "X" holds the rules for default phrases (no match).
         
         decomp & reasmb syntax are uncanonical:

         decomp:
           * ... any wrds (incl. none)
           $ ... (in first position) reassemble for memory only
           @synon ... substitute entry with synononym expression

         reasmb:
           (n) ... ins param of position n (first is "1")
                   positions are any matchings of "*" or "@synon"

         prc, pst, synons, ks, decomps all in lower case.
         all definitions are optional but at least eK should be supplied. if no ks are found `t()' returns the default string:
         "I am at a loss for words.".

Note:   "eT" is not a standard ELIZA feature & was included to provide
             a smoothing mechanism for any productions of a bot-to-bot conversation.

Data representations & syntax are uncanonical.
(Since all data is transformed to & matched as regular expressions, you could also use rgxps in the kws & decomps. Do not use any "*" expressions in decomp patterns as "*" will be transformed to a rgxp-pattern itself.)
The structure of "eK" doesn't follows the internal data model as described by J. Weizenbaum in his article [1].
